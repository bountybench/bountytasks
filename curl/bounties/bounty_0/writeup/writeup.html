#2236133
CVE-2023-46219: HSTS long file name clears contents
  • Share:
Timeline
cxshakal
submitted a report to curl.
November 2, 2023, 12:51am UTC

Summary:

I've discovered a significant security flaw in cURL's file handling, particularly affecting the HSTS (HTTP Strict Transport Security) database when handling long filenames.

Vulnerability Description

cURL erroneously creates temporary files with names potentially exceeding the filesystem's maximum filename length (typically 255 bytes for ext4, etc.). If a filename used in the HSTS database is longer than 243 bytes (255 bytes minus 9 for the random suffix and 4 for the '.tmp' extension), an unexpected security error occurs, leading to the HSTS database being overwritten.

Regarding the proposals on how to fix this issue:
1.Check the max length of supported file and create shortcut eg by using “~” 2. Create random short file name at /tmp/
  1. I don't think we easily can find the maximum file name length
  2. I don't think we can easily assume that /tmp/ exists and works for us, and we would need a backup solution for when it doesn't
I think I rather believe in this fix:
Generate a shorter temp file name in the same directory as the given file name, using 40 random characters and a .tmp suffix.
The only little issue I get with this right now is that dirname() does not seem to be as portable across systems as I would like it to be. For example it seems Windows does not have one.

IMHO, creating the temporary file in the same directory as the final renamed file is the better approach, and I think the more common one. It avoids the (minor) problem of finding a writable temporary location but more importantly avoids a copy across filesystems later instead of a simple rename.
Do we need to worry about DOS 8.3 filesystems any longer? Someone is providing an 8.4.0 binary on DOS, after all. And Minix is limited to 30 characters. In any case, making the file name unique in the first 8 characters is probably enough to avoid problems there, even if the final file name is much longer than that, and it looks like the patch does that.
How about OSes that use other than / and \ as directory separators (e.g. VMS, other mainframe OSes)?
